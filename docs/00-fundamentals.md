# Part 0 — Fundamentals (the tooling underneath)

This doc explains the machinery we lean on: npm, versions, building, the `dist/`
folder, the manifest, and how to actually *read* an extension's code. It's
numbered `00` because it sits underneath everything else — read it whenever the
tooling feels like magic.

---

## 1. What `npm install` does

Command we ran:

```bash
npm install -D vite typescript @types/chrome @crxjs/vite-plugin@latest
```

- **npm** = Node Package Manager, bundled with Node. It downloads reusable
  **packages** from the public **npm registry** (an "app store for JS code").
- `npm install <name>` fetches that package *and everything it depends on* into a
  local folder called **`node_modules/`**. (Our 4 packages pulled in 78 total.)
- It records what you asked for in **`package.json`** and the exact versions in
  **`package-lock.json`**.
- Because those two text files can regenerate `node_modules/` perfectly
  (`npm install` with no args), we **`.gitignore` node_modules** — never commit it.

### The `-D` flag

`-D` = `--save-dev` → records the package under **`devDependencies`** (tools for
*building*) rather than **`dependencies`** (code that ships and runs in the
product). All four of ours are build-time tools, so all are `-D`. None of them
end up inside the finished extension — only our own compiled code does.

### What each package is

| Package | What it is | Why |
|---|---|---|
| `vite` | Build tool + dev server | Compiles TS→JS, bundles, live-reload |
| `typescript` | The TS language + compiler | Typed code; catches errors before runtime |
| `@types/chrome` | Type definitions (no runtime code) | Teaches TS the `chrome.*` API exists |
| `@crxjs/vite-plugin` | A Vite plugin | Makes Vite output a valid Chrome extension |

`@types/*` packages are always dev-only and contain only descriptions of an API,
no running code.

## 2. Version numbers (Semantic Versioning)

Format: **`MAJOR.MINOR.PATCH`**, e.g. `2.7.1`.

- **PATCH** (`1`): bug fixes only — safe to upgrade.
- **MINOR** (`7`): new features, nothing existing breaks — safe to upgrade.
- **MAJOR** (`2`): breaking changes — upgrading may require editing your code.

### Pre-releases (the beta we hit)

`2.0.0-beta.33` is a **pre-release** published *before* stable `2.0.0`. In semver
it sorts *before* `2.0.0`. The maintainers had moved on to stable releases up to
`2.7.1` and abandoned the betas, so `@latest` (= newest **stable**) upgraded us
to `2.7.1`. Newer, stable, maintained.

### The caret `^`

`"^2.7.1"` in package.json means "**≥ 2.7.1 and < 3.0.0**" — accept future minor
and patch updates automatically, but never auto-jump to a breaking major.

Tags: `@latest` = newest stable, `@beta` = newest pre-release, `@1.2.3` = exact.

## 3. Source vs. build — what `dist/` is

Two kinds of files:

- **Source** (`src/`) — what *you write*. TypeScript, comments, human-friendly.
- **Build output** (`dist/`) — what the *tool generates*. Plain JS, bundled,
  browser-ready. `dist` = "distribution".

Chrome can't run `src/` because (a) browsers don't understand TypeScript — types
must be stripped and `.ts` compiled to `.js`; and (b) extensions need files
arranged/named a specific way. **Building** (via Vite + crx) does both and writes
the result into `dist/`.

```bash
npm run build   # → runs "vite build" from package.json "scripts"; writes dist/
```

Consequences:

- **Never hand-edit `dist/`** — the next build overwrites it. Edit `src/`.
- **Never commit `dist/`** — it's reproducible from source (it's in `.gitignore`).
- Weird names like `assets/index.ts-CwVmDuiz.js` contain a **content hash** for
  automatic cache-busting; the generated manifest points at them for you.

### `dev` vs `build`

- `npm run build` — one-shot compile into a clean `dist/`, then stops.
- `npm run dev` — a running process that builds `dist/` **and watches files**,
  rebuilding on every save (live-reload). Load the same `dist/` either way.

`npm run <name>` just executes the matching command in package.json `"scripts"`.

## 4. The manifest — center of gravity

`manifest.json` is the **required blueprint** Chrome reads first. It declares the
name/version, the permissions, and — crucially — **which file runs in which
world** (`background`, `content_scripts`, `action`). Your files are invisible to
Chrome until the manifest points at them.

We have two:

- **`src/manifest.config.ts`** — our source (TypeScript, typo-checked). Edit this.
- **`dist/manifest.json`** — generated by the build; the one Chrome actually
  reads; file paths rewritten to the compiled `assets/*.js`. Never touch it.

## 5. How to read an extension's code

A normal app has ONE entry point read top-to-bottom. An extension has **several**
entry points that start at **different times in response to events**.

**Always start at `src/manifest.config.ts`** — it lists every entry point:

```
manifest.config.ts  ← START HERE
      ├── background:      src/background/index.ts   (the brain)
      ├── content_scripts: src/content/index.ts      (runs in web pages)
      └── action.popup:    src/ui/popup/index.html → popup.ts
```

When each one runs:

| Entry point | Runs when… | Stops when… |
|---|---|---|
| `background/index.ts` | Chrome fires an event it listens for | ~30s idle → Chrome kills it |
| `content/index.ts` | You load a matching web page (fresh per tab) | You leave the page |
| `popup.ts` | You click the toolbar icon | You close the popup |

There is no single "first line." Open a page → content script runs. Click icon →
popup runs. Install → the background `onInstalled` handler runs.

### Recipe for reading any feature

1. Open the manifest → note all entry points (the doors in).
2. Ask "**what event triggers this feature?**" (page load? click? a message?)
3. Find that world's file and its `addListener(...)` for that event — the start.
4. **Follow the messages:** search `sendMessage` / `onMessage` to trace how one
   world asks another to act. (No messages yet — that begins in Part 3.)
